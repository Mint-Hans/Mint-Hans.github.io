我开始学习计算机技术的这一年，恰好是Agent突飞猛进的一年，于此同时，几乎所有的程序员都放弃了手写大量代码，纷纷投入Claude Code、Codex的怀抱，甚至在我写这篇文章之前Claude Code已经帮我写好了一份，其实不错，但博客这种东西我还是喜欢使用自己的文字，文字叙述部分我就自己来。代码不用自己写了，但是工程并不只有写代码，还是有很多需要学习的地方，当然直接问大模型也可以告诉我们还有什么，不过大模型只会傻乎乎地听我们指挥，几步几步地来操作。下面就以游戏开发为例，写一下新的学习范式。

# 第一阶段：建立概念地图（0~3个月）

目标不是会写代码，而是知道世界长什么样。

例如：

**计算机**

* CPU
* 内存
* 线程
* GC
* 编译器
* IDE
* Git

**Unity**

* Scene
* GameObject
* Component
* Prefab
* Script
* Animator
* Input System

**游戏**

* FSM
* Behavior Tree
* Inventory
* Quest
* Combat
* Save

**数学**

* Vector
* Matrix
* Quaternion
* 坐标系

我很感谢引领我成长的学长学姐和那些科普工作者们，以及无数愿意分享自己经验的大神们，我得以较为迅速地建立起对游戏开发知识体系的认识。大模型时代也可以很快地为我们列好地图，建立技术栈。

---

# 第二阶段：让 AI 写，你负责理解

虽然很多人觉得学习阶段还是自己手写的好，我也赞同，但不想写也完全没有问题，省下码字的时间，花在问问题上，是可以学到更多的，是更聪明的做法。

例如：

今天目标：

> 做一个血条。

不要自己硬写。

而是：

> Claude，请实现一个血条。

然后：

逐行阅读。

例如：

```csharp
public class Health : MonoBehaviour
```

问：

> 为什么继承 MonoBehaviour？

继续：

```csharp
[SerializeField]
private Slider hpBar;
```

问：

> SerializeField 为什么不用 public？

继续：

```csharp
public void TakeDamage(int damage)
```

问：

> 为什么不用 float？

不要怕问题幼稚。

AI 最适合回答这种问题。

---

# 第三阶段：不要学代码，要学系统

这是很多人的误区。

很多人的学习是：

```
今天学委托

明天学Lambda

后天学LINQ
```

但是实际上游戏开发更像：

```
今天完成背包系统

明天完成装备系统

后天完成AI系统
```

系统比语法重要得多。

例如：

## 背包系统

你会同时学到：

* List
* Dictionary
* UI
* Drag
* Event
* Serialization
* Save

而不是孤零零学 List。

---

# 第四阶段：所有知识都围绕项目

例如：

今天遇到：

```
Instantiate
```

AI 告诉你：

性能不好。

于是你开始学：

```
GC

对象池

Memory
```

为什么？

因为项目需要。

不是因为教材写到了。

这样知识会牢很多。

---

# 第五阶段：阅读 AI 代码

这是 AI 时代最大的优势。

以前：

每天写：

```csharp
for(...)
```

现在：

每天读：

300 行

500 行

1000 行

优秀代码。

阅读速度会越来越快。

例如：

看到：

```csharp
public class EnemyController
```

你应该一眼知道：

这是：

```
状态

AI

动画

移动

攻击
```

放在哪里。

---

# 第六阶段：不断重构

不要满足于：

AI 能跑。

而是：

```
为什么这里 Update？

为什么不用 Event？

为什么不用 Interface？

为什么不用 State？
```

AI 会陪你讨论。

这是成长最快的时候。

---

# 第七阶段：让 AI 当导师

很多人：

```
给我代码。
```

结束。

我更建议：

```
为什么？

有没有第二种方案？

哪个公司会这样写？

为什么不用 ScriptableObject？

为什么不用 ECS？
```

AI 的价值不是生成。

而是解释。

---

# 第八阶段：开始设计架构

这是很多 AI 用户不会做的。

例如：

不要说：

```
写一个技能系统。
```

而是：

```
技能系统：

Skill

↓

Buff

↓

Cooldown

↓

Animation

↓

Effect

↓

Damage
```

然后：

让 AI 一层层实现。

你负责设计。

---

# 第九阶段：学会 Review AI

以后每天：

Claude：

```
完成。
```

你：

```
Update 太多。

GC 太多。

耦合太高。

命名不好。

拆类。
```

真正的软件工程师越来越像：

Reviewer。

---

# 第十阶段：作品比知识重要

不要追求：

```
今天：

学Rust

学Go

学ECS

学Shader
```

而是：

一年以后：

```
完成：

一个完整Demo

↓

Steam试玩版

↓

发布
```

作品会逼着你：

学习：

* 网络
* UI
* Shader
* 存档
* AI
* Bug
* 性能

成长速度反而最快。

---

# 最后，我认为 AI 时代的软件工程师应该形成这样一种闭环：

```text
提出需求
        ↓
系统设计
        ↓
AI 实现第一版
        ↓
阅读代码
        ↓
理解原理
        ↓
自己修改
        ↓
测试验证
        ↓
发现问题
        ↓
重构优化
        ↓
Git 提交
        ↓
进入下一个功能
```

我甚至想补充一点，也是我认为未来几年最重要的变化：

**不要把自己定位成“代码编写者（Coder）”，而是定位成“软件设计者（Software Designer）”或者“游戏系统设计者（Game Systems Designer）”。**

AI 正在大幅降低“把设计翻译成代码”的成本，但它并不会自动知道你的游戏为什么好玩、系统应该如何演化、哪些功能值得做、哪些架构适合未来扩展。这些仍然需要开发者来决定。

对于你这样的学生，我会给出一个非常具体的目标：

* **学校**：把计算机基础课程学扎实，它们会帮助你判断 AI 的输出是否合理。
* **课外**：每个月完成一个完整的 Unity 游戏系统，而不是每个月学习一种新语言或几十个新概念。
* **长期**：到毕业时，不是以“我会多少技术名词”为目标，而是以“我完成过哪些真正可运行、可维护的项目”为目标。

如果坚持这种学习方式，等你毕业时，你掌握的不仅是某个 AI 工具，而是一套可以适应未来工具变化的开发能力。
